## 2. НАЗНАЧЕНИЕ И ОБЛАСТЬ ПРИМЕНЕНИЯ

**Назначение программы.** «Фамильный Круг» — веб-сервис для создания, хранения и визуализации родословных. Программа позволяет пользователям формировать семейные деревья, вносить биографические сведения о родственниках, фиксировать жизненные события, прикреплять медиаконтент (фотографии, документы), а также совместно работать над деревом в многопользовательском режиме с разграничением прав доступа.

**Область применения.** Программное средство предназначено для:
- частных пользователей, занимающихся составлением родословных и сохранением семейной памяти;
- исследователей, ведущих генеалогическую работу;
- коллективов (семей, исторических сообществ), нуждающихся в совместном редактировании и хранении генеалогических данных.

Работа с программой осуществляется через веб-браузер без установки дополнительного программного обеспечения на компьютер конечного пользователя.

**Основные функциональные возможности:**

| № | Группа функций | Перечень |
|---|---|---|
| 1 | Управление учётными записями | Регистрация, вход, восстановление сессии, редактирование профиля, смена аватара |
| 2 | Управление деревьями | Создание, редактирование, удаление деревьев; многодеревная рабочая среда |
| 3 | Управление персонами | Добавление, редактирование, удаление записей о людях |

| 5 | Визуализация | SVG-граф, режим «Веер», режим «Хронология», сворачивание ветвей |
| 6 | Поиск и анализ | Поиск по имени, вычисление пути родства, статистика дерева |

## 3. ТЕХНИЧЕСКИЕ ХАРАКТЕРИСТИКИ

### 3.1. Постановка задачи

Перед разработкой была сформулирована следующая задача: создать веб-приложение, которое обеспечивает:

1. **Хранение генеалогических данных** — структурированная база данных для хранения сведений о людях, их родственных связях, жизненных событиях и медиаконтенте.

2. **Визуализацию родословной** — интерактивный граф семейного дерева с возможностью навигации (масштабирование, перетаскивание), поиска и сворачивания ветвей.

3. **Многопользовательский доступ** — система ролей (владелец, редактор, наблюдатель, комментатор) и механизм приглашений для совместной работы над деревом.

4. **Совместимость с отраслевым стандартом** — поддержка формата GEDCOM 5.5.1 для обмена данными с другими генеалогическими системами.

5. **Безопасность и аутентификацию** — stateless JWT-аутентификация без хранения серверных сессий.

6. **Масштабируемость развёртывания** — контейнеризация через Docker Compose для единообразного запуска всего стека.

**Ограничения и допущения:**
- Интерфейс программы реализован только на русском языке.
- Генерация схемы базы данных выполняется автоматически средствами Hibernate (`ddl-auto: update`); Flyway-миграции отключены.
- Реализован режим MVP: функции совместного редактирования в реальном времени через WebSocket в данной версии отсутствуют; история изменений ведётся через аудит-лог.

### 3.2. Описание применяемых алгоритмов

#### 3.2.1. Алгоритм вычисления пути родства (BFS)

Для нахождения цепочки родственных связей между двумя персонами используется алгоритм обхода графа в ширину (Breadth-First Search, BFS), реализованный на стороне клиента в файле `frontend/src/utils/findRelationPath.js`.

**Описание алгоритма:**

1. Построить граф смежности: для каждой персоны собрать список её непосредственных соседей (родители, дети, супруги, опекуны).
2. Инициализировать очередь с начальной персоной; отметить её как посещённую.
3. На каждой итерации извлекать персону из очереди и проверять всех её соседей:
   - если сосед является целевой персоной — восстановить путь через массив предков и завершить поиск;
   - если сосед не посещён — добавить в очередь и отметить посещённым.
4. Если очередь опустела, а цель не достигнута — вернуть «путь не найден».

**Сложность:** O(V + E), где V — количество персон, E — количество родственных связей в дереве.

#### 3.2.2. Алгоритм расстановки узлов графа (Genealogy Layout)

Для автоматического позиционирования карточек персон в SVG-пространстве реализован собственный алгоритм иерархической расстановки, описанный в файле `frontend/src/utils/genealogyLayout.js`.

**Параметры расстановки:**

| Константа | Значение | Назначение |
|---|---|---|
| `CARD_W` | 155 px | Ширина карточки персоны |
| `CARD_H` | 180 px | Высота карточки персоны |
| `H_GAP` | 60 px | Горизонтальный зазор между карточками |
| `COUPLE_GAP` | 28 px | Зазор между карточками супругов |
| `V_GAP` | 80 px | Вертикальный зазор между поколениями |

**Описание алгоритма:**

1. Определить корневые узлы дерева (персоны без родителей в системе).
2. Обойти граф в глубину (DFS), присвоив каждой персоне уровень поколения.
3. Сгруппировать супругов: пара размещается рядом с зазором `COUPLE_GAP`; их общие дети центрируются под парой.
4. Разрешить конфликты позиций: если ветви перекрываются по горизонтали, сдвинуть правую ветвь на необходимый отступ.
5. Рассчитать абсолютные координаты (x, y) каждого узла на основании уровня поколения и горизонтальной позиции в ряду.

**Режим «Веер»:** персоны располагаются радиально относительно выбранного центрального предка; угол каждого узла рассчитывается пропорционально его позиции в ряду поколения.

**Режим «Хронология»:** персоны упорядочиваются по оси X согласно году рождения; ось Y фиксирована по поколению.

#### 3.2.3. Алгоритм сворачивания ветвей

Скрытие дочерних узлов реализовано функцией `computeHidden()` в `frontend/src/components/tree/TreeGraph.jsx`:

1. Хранить множество `collapsedIds` — идентификаторы персон, для которых пользователь нажал кнопку сворачивания.
2. При каждом рендере для каждого узла из `collapsedIds` рекурсивно собирать все дочерние идентификаторы в множество `hiddenIds`.
3. При отрисовке SVG пропускать узлы и рёбра, идентификатор которых входит в `hiddenIds`.

### 3.3. Описание структуры программы и технических средств

#### 3.3.1. Общая архитектура

Программа реализована по клиент-серверной трёхзвенной архитектуре:

```
Браузер (React SPA)
        ↕  HTTP/HTTPS
     Nginx (обратный прокси)
      ↙              ↘
Backend (Spring Boot)   MinIO (файловое хранилище)
        ↕
    PostgreSQL
        ↕
      Redis
```

Все компоненты упакованы в Docker-образы и запускаются через `docker-compose.yml`.

#### 3.3.2. Серверная часть (Backend)

**Платформа:** Java 17, Spring Boot 3.x  
**Корневой пакет:** `com.example.backend`

Серверная часть организована по доменному принципу. Каждый домен содержит слои:

| Слой | Пакет | Назначение |
|---|---|---|
| HTTP-слой | `api/` | `@RestController` — приём и возврат HTTP-запросов |
| Бизнес-логика | `application/` | `@Service` — обработка данных, вызов репозиториев |
| Сущности | `entity/` | JPA `@Entity` — ORM-маппинг таблиц БД |
| DTO | `dto/` | Объекты для сериализации запросов и ответов |
| Репозитории | `repository/` | Spring Data JPA — запросы к базе данных |

**Доменные модули:**

| Модуль | Ключевые сущности | Назначение |
|---|---|---|
| `auth` | `User` | JWT-аутентификация, регистрация, профиль |
| `person` | `Person` | CRUD персон семейного дерева |
| `relationship` | `Relationship` | Родственные связи четырёх типов |
| `event` | `Event`, `EventPerson` | Жизненные события персон |
| `comment` | `Comment` | Комментарии к любой записи |
| `tree` | `FamilyTree`, `TreeMember` | Деревья и участники с ролями |
| `tree/invite` | `InviteToken`, `TreeInvite` | Токен-приглашения по электронной почте |
| `media` | `Media` | Загрузка и хранение медиафайлов через MinIO |
| `graph` | — | Вспомогательные алгоритмы визуализации (серверная часть) |
| `gedcom` | — | Парсер и генератор GEDCOM 5.5.1 |
| `audit` | `AuditLog` | Журнал изменений |
| `shared` | `BaseEntity` | Глобальные исключения, конфиги MinIO и почты |

**Безопасность:**
- `JwtTokenProvider` — генерация и валидация JWT.
- `JwtAuthenticationFilter` — фильтр Spring Security, добавляемый перед стандартной цепочкой.
- Публичные эндпоинты: `/api/auth/**`, `/api/ping`, `/api/invites/**`.
- Все остальные маршруты требуют действительного Bearer-токена.

**База данных:** PostgreSQL 15. Схема создаётся автоматически Hibernate (`ddl-auto: update`). Соединение через Spring Data JPA / Hibernate ORM.

**Кеш:** Redis 7 — используется для хранения вспомогательных данных и сессий токенов.

**Файловое хранилище:** MinIO (S3-совместимый объектный стор). Файлы сохраняются в бакет `family-tree-media`; URL для доступа проксируется через Nginx по пути `/media/`.

### 3.4. Обоснование принятых технических решений

#### 3.4.1. Выбор языка и фреймворка серверной части

**Java 17 + Spring Boot 3** выбраны по следующим причинам:
- зрелая экосистема с широким набором готовых интеграций (Spring Security, Spring Data JPA, Spring Mail);
- строгая типизация и встроенные инструменты рефакторинга снижают вероятность ошибок;
- поддержка аннотаций Lombok исключает необходимость написания шаблонного кода (геттеры, сеттеры, конструкторы);
- отраслевой стандарт для enterprise-разработки, хорошо знакомый целевой аудитории (студенты программной инженерии).

#### 3.4.2. Выбор СУБД

**PostgreSQL 15** выбран как основная СУБД по следующим причинам:
- надёжная поддержка транзакций (ACID);
- возможность хранения структурированных отношений «многие ко многим» (родственные связи, участники дерева);
- встроенная поддержка JSONB для хранения произвольных атрибутов;
- хорошая интеграция с Hibernate/Spring Data JPA.

**Redis** используется как вспомогательный кеш: высокая скорость чтения/записи в памяти подходит для хранения краткосрочных данных.

#### 3.4.3. Выбор объектного хранилища

**MinIO** выбран вместо хранения файлов в СУБД по следующим причинам:
- хранение бинарных файлов в реляционной БД снижает производительность;
- S3-совместимый API даёт возможность в будущем перейти на облачный хостинг (AWS S3, Yandex Object Storage) без изменения кода;
- MinIO легко развёртывается локально в Docker-контейнере.

#### 3.4.4. Выбор технологий клиентской части

**React 19** выбран как основная UI-библиотека:
- компонентный подход упрощает переиспользование элементов интерфейса;
- React Router 7 обеспечивает клиентскую маршрутизацию (SPA) без перезагрузки страницы;
- большое сообщество и богатая экосистема пакетов.

**Vite 7** используется как инструмент сборки:
- значительно быстрее Webpack при разработке (HMR на основе ESM);
- минимальная конфигурация «из коробки».

**Plain JSX (без TypeScript)** — принципиальный выбор для снижения порога вхождения и ускорения разработки MVP. TypeScript-зависимости присутствуют в `devDependencies` для возможной миграции в будущем.

**SVG-граф** реализован без сторонних графовых библиотек (D3.js, vis-network) — это исключает избыточные зависимости и обеспечивает полный контроль над отрисовкой. Расчёт координат узлов выполняется собственным алгоритмом (`genealogyLayout.js`).

**CSS без фреймворков** (без Tailwind, styled-components) — единый файл `index.css` (~1800 строк) с CSS-переменными даёт полный контроль над стилями и поддержкой тёмной темы без дополнительных зависимостей.

#### 3.4.5. Выбор формата обмена данными

**GEDCOM 5.5.1** — международный открытый стандарт генеалогических данных, поддерживаемый всеми крупными генеалогическими платформами (Ancestry, FamilySearch, MyHeritage). Поддержка GEDCOM обеспечивает совместимость и возможность миграции данных.

**JSON-экспорт** — собственный формат, включающий полный снимок дерева (персоны, связи, события, медиа) для резервного копирования и интеграции со сторонними инструментами.

#### 3.4.6. Аутентификация

**Stateless JWT** выбран вместо серверных сессий:
- не требует хранилища сессий (горизонтальное масштабирование упрощается);
- токен содержит всю необходимую информацию о пользователе (subject, expiration);
- стандартный подход для REST API.

#### 3.4.7. Контейнеризация

**Docker Compose** обеспечивает воспроизводимую среду: одна команда `docker compose up` разворачивает весь стек (6 сервисов) в согласованном состоянии. Это исключает расхождения «работает на моей машине» и упрощает проверку работоспособности.

### 3.5. Описание организации входных и выходных данных

#### 3.5.1. Входные данные

Входные данные поступают в систему из следующих источников:

**1. HTTP-запросы от клиентской части**

Все запросы направляются на `/api/**`. Тело запросов передаётся в формате JSON; файлы — в формате `multipart/form-data`.

| Тип данных | Формат | Пример эндпоинта |
|---|---|---|
| Учётные данные | JSON `{email, password, name}` | `POST /api/auth/login` |
| Данные персоны | JSON `{firstName, lastName, gender, birthDate, ...}` | `POST /api/trees/{id}/persons` |
| Родственная связь | JSON `{fromPersonId, toPersonId, type}` | `POST /api/relationships` |
| Жизненное событие | JSON `{type, date, description}` | `POST /api/events` |
| Комментарий | JSON `{entityType, entityId, text}` | `POST /api/comments` |
| Медиафайл | `multipart/form-data` (файл + метаданные) | `POST /api/upload/photo` |
| GEDCOM-файл | `multipart/form-data` (текстовый файл `.ged`) | `POST /api/gedcom/import` |
| Данные приглашения | JSON `{email, role, treeId}` | `POST /api/invites` |

**2. Пользовательский ввод в форме браузера**

Данные вводятся через HTML-формы в интерфейсе и передаются клиентской частью в виде JSON через API-слой.

**3. Внешние файлы**

- Файлы GEDCOM (.ged) — стандарт 5.5.1, кодировка UTF-8 или ANSEL.
- Медиафайлы: фотографии (JPEG, PNG, WEBP ≤ 10 МБ), документы (PDF, DOC, DOCX, XLS, XLSX, TXT ≤ 10 МБ).

#### 3.5.2. Выходные данные

| Тип выходных данных | Формат | Описание |
|---|---|---|
| JWT-токен | Строка (Bearer) | Возвращается при успешной аутентификации |
| Список деревьев | JSON-массив | Отображается на странице «Мои деревья» |
| Данные персоны | JSON-объект | Отображаются в панели персоны |
| SVG-граф | SVG-разметка | Визуализация родственных связей в браузере |
| Путь родства | JSON-массив шагов | Преобразуется в текстовое описание на клиенте |
| Статистика дерева | JSON-объект | Отображается в модальном диалоге |
| GEDCOM-файл | Текстовый файл `.ged` | Скачивается в браузере при экспорте |
| JSON-файл дерева | Текстовый файл `.json` | Скачивается в браузере при экспорте |
| Аудит-лог | JSON-массив записей | Отображается в диалоге «Журнал изменений» |
| URL медиафайла | Строка | Используется для отображения изображений через MinIO |
| Сообщение об ошибке | JSON `{message}` / Toast | Отображается при нарушении ограничений |

#### 3.5.3. Схема потоков данных

```
Пользователь (браузер)
    │
    ▼
[React SPA] ──fetch──► [Nginx :80]
                            │
                   ┌────────┴────────┐
                   ▼                 ▼
           [Spring Boot :8080]   [MinIO :9000]
                   │
          ┌────────┴────────┐
          ▼                 ▼
    [PostgreSQL :5432]  [Redis :6379]
```
